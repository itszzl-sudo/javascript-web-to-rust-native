# JavaScript-Web-to-Rust-Native 实现路线图

## 项目概览

JavaScript-Web-to-Rust-Native 是一个创新项目，旨在将任意打包后的 JavaScript 代码编译为原生 Rust 二进制程序。本路线图规划了从概念验证到生产就绪的完整开发路径。

---

## 核心原则

> **"浏览器不关心你的源码是什么框架，只关心最终能否执行 JavaScript"**

这意味着：
- ✅ 不需要处理 Vue SFC / React JSX / Svelte 模板
- ✅ 不需要处理 TypeScript 编译
- ✅ 只需要处理**打包后的标准 JavaScript**
- ✅ 只需要处理 **DOM/BOM API 调用**

## 待讨论方案

### 4.1. Registry 架构方案

**背景**：新增 Registry 用来为 Director 提供接口和数据，Director 可能会被有更好的方案替代，我们重心在 runtime 和 translator。

**方案描述**：

| 组件 | 职责 | 状态 | 备注 |
|------|------|------|------|
| **Registry** | 为 Director 提供统一的接口和数据管理 | ⏳ 待讨论 | 可能包含事件处理器注册表、DOM快照管理、配置管理等 |
| **Director** | 协调 jrust 实例和事件分发 | ⏳ 待评估 | 可能有更好的方案替代 |
| **jrust-runtime** | 核心运行时，提供 DOM/BOM 支持 | ✅ 优先 | 重点投入 |
| **jrust-translator** | JavaScript 转译器 | ✅ 优先 | 重点投入 |

**讨论要点**：
1. Registry 是否需要独立存在，还是可以合并到 runtime 中？
2. Director 的职责是否可以由 runtime 直接承担？
3. Registry 与 runtime/translator 的接口定义？
4. 是否需要单独的配置管理模块？

**推荐策略**：
- 优先完善 **jrust-runtime** 和 **jrust-translator**
- Registry 作为可选增强功能，视需求决定是否实现
- Director 保持精简，仅保留核心协调能力

## 结语

JavaScript-Web-to-Rust-Native 的核心理念是**统一处理**：不管你的源码是什么框架，打包后都是标准 JavaScript，我们的转译器统一处理这些 JavaScript 代码。

**核心成功因素**：
1. 坚持"不关心框架，只关心 JavaScript"的原则
2. 团队的稳定性和专业性
3. 持续的社区建设
4. 严格的质量保证

让我们共同努力，打造真正的通用 JavaScript 原生编译平台！
